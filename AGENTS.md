# Unity 1인칭 3D PvP 전투 게임 초기 기반 구현 지침

Unity 6 Universal 3D 프로젝트에서 1인칭 3D 전투 게임의 초기 기반을 만든다.

이 게임은 최종적으로 온라인 PvP 멀티플레이를 목표로 한다.
다만 초기 단계에서는 네트워크를 구현하지 않는다.
지금은 앞으로 전투, 스킬, 봇, 건물, UI, 게임 상태, 네트워크를 확장하기 쉬운 작고 안정적인 3D 조작 샌드박스를 만드는 것이 목표다.

초반 구현은 오프라인 로컬 플레이어 1명만 대상으로 하되, 구조는 여러 PlayerEntity를 지원하고 나중에 서버 권한 멀티플레이로 전환하기 쉽게 설계한다.

## 최우선 개발 원칙

가장 중요한 원칙은 최적화와 구조 유지다.

버그나 수정 요청이 있을 때 증상만 막는 임시 코드를 덧붙이지 말고, 반드시 기존 구조의 원인, 중복, 병목, 충돌을 먼저 찾는다.

수정할 때는 "이번 위기만 넘기는 코드"가 아니라, 전체적인 틀 안에서 장기적으로 유지 가능한 방식으로 고친다.

## 수정 원칙

- 사용자가 수정 요청을 하면 바로 새 코드를 덧붙이지 않는다.
- 먼저 관련 코드 흐름을 읽고 원인을 파악한다.
- 기존 구조 안에서 해결할 수 있으면 기존 코드를 정리해서 해결한다.
- 같은 역할의 코드가 중복되면 통합한다.
- 임시 플래그, 예외 처리, 하드코딩으로 상황만 넘기지 않는다.
- 필수 기능이 완전히 누락된 경우에만 새 코드를 추가한다.
- 새 코드를 추가할 때도 기존 시스템의 책임 범위 안에 둔다.
- 변경 후 성능, GC, Update 호출 수, 물리 처리 비용, 할당 발생 가능성을 고려한다.
- 매 프레임 불필요한 Find, GetComponent 반복, LINQ, new 할당, 문자열 생성, 전체 객체 탐색을 피한다.
- 관련 없는 리팩토링은 하지 않는다.
- 기존 사용자 변경사항을 되돌리지 않는다.

## 멀티플레이 전제

최종 목표는 온라인 PvP 멀티플레이이다.
이번 단계에서는 네트워크 구현을 하지 않지만, 처음부터 멀티플레이 전환을 고려한다.

- 로컬 입력 처리와 캐릭터 시뮬레이션을 분리한다.
- 카메라와 UI는 로컬 플레이어 전용으로 분리한다.
- 캐릭터 상태는 PlayerId, TeamId, Position, Velocity, Health 같은 데이터 중심으로 확장 가능하게 둔다.
- 이동, 전투, 스킬은 나중에 서버 권한 방식으로 옮기기 쉽게 만든다.
- 공격 버튼 입력과 실제 공격 판정을 분리할 수 있는 구조를 고려한다.
- 클라이언트는 최종적으로 입력/요청을 보내고, 서버가 판정하는 구조를 목표로 한다.
- 지금은 오프라인 로컬 플레이어 1명만 구현하되, 구조는 여러 PlayerEntity를 지원할 수 있게 한다.
- Netcode for GameObjects, Mirror, FishNet 같은 네트워크 솔루션을 나중에 붙일 수 있게 강한 결합을 피한다.
- 카메라, 입력, HUD는 LocalPlayer에만 연결되는 구조로 둔다.
- 캐릭터 이동 로직은 가능한 한 예측 가능하고 재사용 가능한 시뮬레이션 코드로 작성한다.

## 구조 원칙

입력, 이동, 카메라, 전투, 스킬, 능력치, 봇, 건물, UI, 게임 상태, 네트워크 어댑터를 분리한다.

각 시스템은 자기 책임만 가진다.

- Input: 입력 읽기만 담당
- Command: 입력을 캐릭터 명령으로 변환
- Movement: 이동, 점프, 앉기, 중력 처리
- Camera: 로컬 플레이어 시점 회전과 카메라 높이 처리
- Combat: 공격 판정, 피해 처리
- Ability: 스킬 사용과 쿨타임
- Stats: 체력, 방어력, 이동속도 등 수치 계산
- Bot: 목표 선택과 행동 결정
- Building: 포탑, 넥서스 규칙
- UI: 화면 표시만 담당
- GameState: 시작, 사망, 부활, 승패 등 상태 관리
- Networking: 나중에 네트워크 솔루션을 연결할 어댑터 역할

초기 단계에서는 Input, Command, Movement, Camera, UI, Config만 구현한다.
Combat, Ability, Stats, Bot, Building, GameState, Networking은 아직 구현하지 말고, 나중에 붙일 수 있도록 구조만 고려한다.

## 데이터 원칙

- 이동속도, 점프높이, 중력, 마우스 감도, 앉기 높이 같은 수치는 코드 안에 흩뿌리지 않는다.
- 중앙 설정 데이터 또는 ScriptableObject에서 관리한다.
- 향후 체력, 피해량, 쿨타임, 사거리, 아이템, 버프, 디버프 수정치를 합산할 수 있도록 기본값 + 수정치 구조를 고려한다.
- 매직 넘버를 피하고 의미 있는 이름의 설정값으로 둔다.
- 나중에 서버와 클라이언트가 같은 수치를 참조할 수 있도록 설정 데이터의 책임을 명확히 둔다.

## 최적화 원칙

- 단일 게임 루프 흐름을 유지한다.
- 매 프레임 전체 객체 탐색을 피한다.
- 필요하면 캐시, 레이어 마스크, 거리 필터, 공간 분할, 오브젝트 풀링을 사용한다.
- 투사체, 이펙트, 피해 숫자, 임시 오브젝트는 나중에 풀링할 수 있게 설계한다.
- Update는 필요한 컴포넌트에만 둔다.
- Unity Garbage Collector가 자주 돌지 않도록 런타임 할당을 줄인다.
- Physics 쿼리는 LayerMask와 NonAlloc API를 우선 고려한다.
- UI 갱신은 매 프레임이 아니라 값이 변했을 때 또는 제한된 주기로 한다.
- 네트워크 전환을 고려해 이동/상태 데이터는 필요 이상으로 무겁게 만들지 않는다.

## Unity 구현 기준

- Unity 기본 기능을 우선 사용한다.
- 과한 외부 패키지는 추가하지 않는다.
- CharacterController를 사용한다.
- 입력은 일단 Unity 기본 Input 또는 명확히 설명 가능한 방식으로 구현한다.
- 씬/프리팹 연결이 필요한 부분은 단계별로 알려준다.
- 코드만으로 자동 생성 가능한 오브젝트는 GameBootstrap에서 최소한으로 생성해도 된다.
- 단, 장기적으로는 프리팹화하기 쉬운 구조로 둔다.
- LocalPlayerController는 입력과 카메라를 소유한다.
- PlayerEntity는 캐릭터 상태와 이동 대상 역할을 한다.
- 나중에 원격 플레이어는 PlayerEntity만 있고 LocalPlayerController는 없는 형태로 확장 가능해야 한다.

## AI 작업 방식

작업 전:
- 관련 파일을 먼저 읽는다.
- 현재 구조를 짧게 요약한다.
- 어떤 파일을 만들거나 바꿀지 정한다.

작업 중:
- 관련 없는 리팩토링을 하지 않는다.
- 큰 구조를 바꿀 때는 이유를 설명한다.
- 기존 사용자 변경사항을 되돌리지 않는다.
- 임시방편 대신 원인 기반으로 수정한다.
- 새 기능이 구조를 오염시키면 먼저 더 좋은 구조를 제안한다.

작업 후:
- 변경 내용을 짧게 요약한다.
- Unity에서 어떤 GameObject에 어떤 컴포넌트를 붙여야 하는지 설명한다.
- SampleScene에서 테스트하는 방법을 알려준다.
- 컴파일 에러 가능성이 있는 부분을 확인한다.
- 다음 단계로 무엇을 구현하면 좋을지 제안한다.

## 장기 구현 방향

이 프로젝트는 다음 순서로 확장한다.

1. 1인칭 이동/카메라
2. 중앙 설정 데이터
3. PlayerEntity와 로컬/원격 플레이어 구조
4. 체력/피해 시스템
5. 기본 공격
6. 직업 구조
7. 스킬 구조
8. 건물 구조
9. 봇 AI
10. UI
11. 최적화/풀링/디버그
12. 네트워크 솔루션 선택
13. 서버 권한 이동/전투 구조
14. 온라인 PvP 테스트

구조가 잡힌 뒤 Unity MCP나 Unity 내 AI 도구를 연결해서 씬 오브젝트, 프리팹, Inspector 연결, UI 배치 같은 에디터 작업을 자동화한다.

## 네트워크 방향성

최종 PvP 구조는 서버 권한 방식을 목표로 한다.

- 클라이언트는 입력을 보낸다.
- 서버는 이동, 공격, 피해, 사망, 승패를 판정한다.
- 클라이언트는 서버 상태를 받아 보간/표시한다.
- 공격 판정과 피해 확정은 최종적으로 서버가 담당한다.
- 클라이언트 단독 판정은 프로토타입 단계에서만 허용한다.
- 네트워크 솔루션은 추후 Netcode for GameObjects, Mirror, FishNet 중 프로젝트 상황에 맞게 선택한다.

## 핵심 문장

증상만 막는 패치가 아니라, 기존 구조의 원인·중복·병목을 찾아 최적화와 전체 구조를 해치지 않는 방식으로 수정하라.
